# Darkness Kingdom Playable Loop Milestone

## Summary
Build the next major milestone as a playable vertical slice: `MainHub -> Campaign tab chapter map -> Formation -> Battle -> Rewards/Unlocks -> Return to MainHub`. Work directly on `main`, as requested, but create a backup branch from the starting commit before changes and commit only after checks pass.

Grounding sources inspected: [README](https://github.com/Ezz10099/Darkness-kingdom/blob/main/README.md), [Design Constitution](https://github.com/Ezz10099/Darkness-kingdom/blob/main/docs/DARKNESS_KINGDOM_DESIGN_CONSTITUTION.md), [Full Game Design](https://github.com/Ezz10099/Darkness-kingdom/blob/main/docs/DARKNESS_KINGDOMS_FULL_GAME_DESIGN.md), [UI Style Guide](https://github.com/Ezz10099/Darkness-kingdom/blob/main/docs/UI-Style.md.txt), [Battle Asset Architecture](https://github.com/Ezz10099/Darkness-kingdom/blob/main/docs/battle-asset-architecture.md), [profit roadmap](https://github.com/Ezz10099/Darkness-kingdom/blob/main/docs/profit-without-p2w-roadmap.md), and stale open [PR #79](https://github.com/Ezz10099/Darkness-kingdom/pull/79).

## Implementation Changes
- Main Hub shell: keep `MainHubScene` as the persistent controller with fixed top bar, center content, side shortcuts, and bottom nav: `Campaign | Heroes | Summon | Guild | More`.
- Use existing shell assets where available: `mainHubBackground`, `topBarFrame`, `bottomNavFrame`, `panelLarge`, `panelSmall`, `buttonPrimary`, `buttonSecondary`, `buttonDanger`, and nav icons.
- Generate only missing milestone-critical UI art: `icon_more.png`, campaign stage node states, and the Phase 1 battle button/card/circle frames listed in `docs/battle-asset-architecture.md`.
- All generated assets must contain no baked text. Render labels in Phaser. Save project-bound generated PNGs into repo asset folders and register them in `assetManifest.js` only after the files exist.
- Campaign tab: replace the current abstract panel map with a chapter-map view using existing chapter backgrounds, generated node assets, cleared/current/locked/boss states, selected stage details, reward preview, and a clear Battle/Formation call to action.
- Campaign flow: selecting the current unlocked stage opens formation, `BACK` returns to `MainHub` Campaign tab, and victory returns to the next focused stage in the hub.
- Formation rules: standardize to 5 heroes with `FRONT max 2` and `BACK max 3`; update `GameState` normalization, formation editor validation, and battle rendering so no selected unit is silently dropped.
- Battle readiness: set `DEBUG_BATTLE_SLOT_CALIBRATION` to `false` by default or gate it behind an explicit dev flag; no debug markers or slot printouts in normal play.
- Battle rendering: include `heroDefId` on player combatants in `BattleEngine`; use hero battle PNGs where available; keep safe rectangle fallback only for genuinely missing textures.
- Battle controls: keep Begin Battle gating, Pause, x1/x2, Auto, Settings, and ultimate buttons, but render them with the battle UI asset frames and ensure they do not overlap on 480x854.
- Monetization/analytics work is out of scope for this milestone except preserving economy behavior and not breaking the profit-roadmap guardrails.

## Automation Workflow
- Direct-main cadence: one commit per checkpoint, each pushed only after `npm run check` passes locally.
- Suggested commits: `chore: clean identity and safety checks`, `feat: add playable-loop UI assets`, `feat: upgrade main hub campaign map`, `fix: stabilize formation and battle loop`, `test: add playable loop smoke coverage`.
- User supervision checkpoints happen after each pushed commit with a short status: changed files, tests run, screenshots if browser verification is available, and next planned commit.
- Stop immediately for user review if checks fail, generated assets look off-brand, a save migration is ambiguous, GitHub rejects direct-main push, or browser smoke testing cannot be performed after UI changes.

## Test Plan
- Update `npm run lint` to syntax-check all app JS under `main.js`, `src/data`, `src/systems`, `src/scenes`, `src/ui`, and `src/debug`.
- Replace the placeholder `npm test` with dependency-free `node --test` smoke tests.
- Add tests for: asset manifest paths exist, 80 campaign stages across 5 regions, hero definitions have asset bundle coverage, formation normalization enforces 2 front/3 back, no normal source/docs contain forbidden old identity terms outside approved hidden migration constants.
- Manual/browser smoke: first-session load, onboarding into MainHub, Campaign tab selection, stage 1-1 formation, Begin Battle, Pause/Speed/Auto toggles, victory reward, unlock handling, save/reload, and return-to-hub focus.
- Final acceptance: no visible old identity, no default debug overlay, current stage can be played end to end, all registered assets load, and GitHub Actions passes on `main`.

## Assumptions
- Direct-main means no PR workflow, but backup branch, checkpoint commits, and user review pauses are still required.
- Generated art is allowed for missing UI assets, using built-in image generation first; transparent-style assets should use chroma-key removal if needed.
- Existing hero and campaign PNGs should be reused; do not regenerate hero art unless separately requested.
- Save compatibility should be preserved through hidden migrations, while all new public names use Darkness Kingdom terminology.
